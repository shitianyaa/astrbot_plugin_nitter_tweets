## Issue #30 代理配置调研

- 原始要求：查看 GitHub Issue #30，并补充常见代理协议及账号密码认证方式。
- Issue 状态：`OPEN`；用户希望在国内网络环境下通过代理访问公共 Nitter 实例。
- 当前状态：已完成需求和现有代码路径调研，尚未修改业务代码。
- 现有实现：Nitter RSS 和媒体请求均通过 `media_support/network.py` 的 `compat_urlopen()` 调用 Python `urllib.request.urlopen`；插件配置中没有代理项。
- 协议范围：常见类型包括 HTTP、HTTPS、SOCKS4/SOCKS4a、SOCKS5 和 SOCKS5h。Issue 的实用支持范围至少应考虑 `http://`、`https://`、`socks5://`、`socks5h://`。
- 行为说明：HTTP 代理可以通过 `CONNECT` 隧道访问 HTTPS 目标，代理地址不必因为 Nitter 使用 HTTPS 而写成 `https://`。
- 认证范围：代理可能不需要认证，也可能使用用户名和密码；HTTP 企业代理还可能使用 Digest、NTLM 或 Kerberos。SOCKS5 常见用户名/密码认证。
- 安全要求：代理密码不得写入日志、错误消息、文档示例或诊断输出；WebUI 应使用密码字段；用户名和密码中的特殊字符需要正确编码或通过独立字段传递。
- 初步设计建议：将代理类型、主机、端口、用户名和密码拆分配置，避免在普通 URL 字段直接展示凭据。只代理 Nitter RSS，是否同时代理 xdown 和媒体下载需另行明确。
- 兼容性结论：标准库 `ProxyHandler` 可处理 HTTP/HTTPS 代理；SOCKS5/SOCKS5h 需要额外 SOCKS 支持或更换网络客户端。
- 已查资料：Python `urllib.request.ProxyHandler` 文档、curl 代理协议说明、RFC 9110 `CONNECT` 规范。
- 未解决问题：最终支持哪些认证机制；是否新增生产依赖；代理作用域是仅 RSS 还是全部外部媒体请求。
